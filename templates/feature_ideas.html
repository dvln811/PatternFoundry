Feature Ideas & Design

Running design document for upcoming features — gamification, stats, challenges, and engagement mechanics.

1. Iron Man Mode Live

Concept

A persistent account challenge that simulates prop firm evaluation conditions. The user commits to a funded-account-style challenge where they cannot reset, cherry-pick sessions, or quit without consequence. Designed to build discipline and replicate the psychological pressure of real capital at risk.

Core Rules

  • Configurable starting capital — default $10,000, user selects at challenge start ($5k / $10k / $25k / $50k)
  • Target — reach +10% of starting capital to "pass" the challenge
  • Drawdown limit — trailing max drawdown of 20% from peak equity (measured from starting balance). Breach = instant fail
  • No session skipping — sessions assigned randomly (character + seed), user must complete or it counts as flat
  • No cherry-picking — no instrument/character selection during a challenge
  • Forfeit penalty — abandoning mid-challenge ends the run, recorded as forfeit in history
  • Separate tracking — Iron Man stats never mix with free practice stats
  • Multiple attempts — unlimited retries, each tracked as "Attempt #N"

Configurable Parameters

User-Selected at Start

  • Starting balance: $5,000 / $10,000 / $25,000 / $50,000
  • Difficulty preset (future): Bronze (+10%, -20%), Silver (+8%, -10%), Gold (+6%, -5%)

System Constants (v1)

  • Target: +10% from start
  • Max drawdown: -20% trailing from peak (relative to start balance)
  • Position sizing: user-controlled but capped at reasonable % of challenge balance

Session Flow (when Iron Man active)

  1. User opens Simulator → system detects active Iron Man run
  2. Character and seed assigned randomly (no user choice)
  3. Session plays normally — all trading tools available
  4. On session complete (all ticks played): results auto-submitted to /api/ironman/session-complete
  5. Balance, peak, drawdown recalculated server-side
  6. If target or drawdown breached → challenge ends automatically
  7. If user navigates away mid-session → counts as flat day (no trades recorded)

Completion Outcomes

  • Target Reached — "Passed" badge, best metrics recorded (days taken, win rate, max drawdown touched)
  • Drawdown Breach — "Failed" with stats showing where it went wrong
  • Forfeit — voluntary quit, recorded but no penalty beyond losing progress

API (Implemented)

GET /api/ironman/status — current run or history

POST /api/ironman/start — begin new challenge

POST /api/ironman/session-complete — submit session results (pnl, trades, wins, character, seed)

POST /api/ironman/forfeit — end active run

2. Session History & Equity Curve Live

Concept

Automatic journaling without manual effort. Every completed simulator session is recorded with key metrics, building a long-term performance picture. The "journal that writes itself."

What's Tracked (per session)

  • Date, character/instrument, number of trades, wins, losses
  • Session P/L (dollar amount)
  • Cumulative equity curve across all sessions

Display

  • Summary cards: total sessions, total trades, overall win rate, cumulative P/L
  • Equity curve: LightweightCharts area series showing running P/L
  • Session history table: sortable, most recent first

TODO — Server-Side Migration

  • Currently localStorage only — need to migrate to server-side (SQLite session_history table)
  • API: POST /api/sessions/record and GET /api/sessions/history
  • Sync on page load, persist across devices

3. Practice Streaks & Calendar Heatmap Live

Concept

Duolingo-style streak counter that rewards consistency. The goal is habit formation — making daily practice automatic. Combined with a GitHub-style contribution heatmap for visual satisfaction.

Mechanics

  • Streak counter — consecutive days with at least 1 completed session
  • Streak continues if today or yesterday has activity (grace for timezone drift)
  • Calendar heatmap — 53 weeks (371 days), intensity by sessions per day
  • Levels: 0 (empty), 1 (light), 2 (medium), 3-4 (bright), 5+ (max)

4. Achievements & Badges Concept

Philosophy

Reward process and discipline, not outcomes. A trader who follows their plan and loses should earn more than one who revenge-trades and gets lucky. Badges are cosmetic — no gameplay advantage.

Proposed Badge Categories

Milestone Badges

  • "First Blood" — complete first ever session
  • "Ten Sessions" / "Fifty" / "Hundred" / "Thousand"
  • "Century Club" — 100 total trades executed
  • "Iron Will" — pass first Iron Man challenge

Discipline Badges

  • "Plan the Trade" — 10 consecutive sessions where SL/TP hit (no manual close)
  • "Flat After a Loss" — did not re-enter within 30 seconds of a losing trade (x5)
  • "Three and Done" — completed 5 sessions with ≤3 trades each (no overtrading)
  • "Survived the Drawdown" — had 5+ consecutive losers in Iron Man, continued and recovered

Streak Badges

  • "Week Warrior" — 7-day streak
  • "Month of Mastery" — 30-day streak
  • "Quarterly Grind" — 90-day streak

Performance Badges (outcome-based, secondary)

  • "Green Week" — 5 consecutive profitable sessions
  • "Sniper" — 80%+ win rate across 10+ trades in a single session
  • "Risk Manager" — avg loss < avg win for 20 consecutive sessions

Display

  • Badge shelf on Stats page (grid of earned badges, dimmed for unearned)
  • Tooltip on hover showing requirements + date earned
  • Optional: badge count shown on avatar/profile

5. Session Grades & Report Cards Concept

Concept

After each session, auto-generate a "report card" grading execution quality — not just P/L. This teaches traders to evaluate their process independent of outcome.

Grading Criteria (proposed)

  • Plan Adherence (40%) — Did SL/TP hit naturally? Or was it manual close / SL moved?
  • Risk Management (25%) — Was position sizing consistent? Was R:R ≥ 1.5:1?
  • Overtrading Score (20%) — Number of trades vs session length. Penalize rapid re-entry after loss
  • Patience (15%) — Time between trades. Reward waiting for setup vs. impulse entries

Grade Scale

  • A+ (95-100) → A (90-94) → B+ (85-89) → B (80-84) → C (70-79) → D (60-69) → F (<60)
  • Displayed as letter grade + numeric score
  • Trend line of grades over time on Stats page

Implementation Notes

  • Requires tracking: time of each trade entry/exit, whether SL/TP was moved, position sizes
  • Server-side calculation after session submit
  • Optional "session notes" text field for minimal manual journaling

6. Challenges & Missions Concept

Concept

Structured constraints that teach specific skills. Rather than "just trade," give the user a focused objective that forces deliberate practice of a specific behavior.

Daily Challenges (rotating)

  • "First Pullback Only" — only trade the first pullback after ORB
  • "Scalper" — take 5+ trades, each held less than 2 minutes
  • "Sniper Session" — maximum 2 trades, both must be profitable
  • "Risk:Reward" — every trade must have at least 2:1 R:R setup
  • "The Wait" — do not enter a trade in the first 15 minutes

Weekly Targets

  • "Consistency Week" — achieve positive P/L on 4 out of 5 sessions
  • "Efficiency" — average ≥2:1 R:R across all trades for the week
  • "Volume" — complete 10 sessions in one week
  • "Process Over Outcome" — earn grade B or higher on 5 sessions

Implementation Notes

  • Challenge completion checked server-side after session submit
  • Completing challenges could award XP or badge progress
  • Challenges should be opt-in (shown on dashboard, click to activate)
  • Don't gate features behind challenges — they're for engagement only

7. Performance Analytics (Advanced Stats) Planned

Metrics to Add

  • Profit Factor — gross wins / gross losses
  • Max Drawdown — largest peak-to-trough decline
  • Average R-Multiple — avg(pnl / risk_per_trade)
  • Expectancy — (win% × avg_win) - (loss% × avg_loss)
  • Best/Worst Trade — single largest win and loss
  • Time-of-Day Performance — P/L broken down by time of entry (morning vs afternoon)
  • Instrument Breakdown — per-character win rate and P/L
  • Consecutive Wins/Losses — max streaks in both directions
  • Average Hold Time — mean time between entry and exit

Visualization

  • R-multiple distribution histogram
  • Win rate by instrument bar chart
  • Drawdown curve (separate from equity curve)
  • Monthly P/L bar chart

8. Visual Flair & Engagement Polish Concept

Ideas

  • Confetti animation — on green session completion or achievement unlock
  • Equity curve "level up" — border color changes at milestones (bronze → silver → gold)
  • Sound effects — subtle audio on trade execution, SL/TP hit (optional, off by default)
  • Streak fire animation — the 🔥 emoji pulses/glows at streak milestones
  • Iron Man completion celebration — full-screen animation on passing the challenge

Principles

  • All visual flair is optional and can be disabled
  • Never block the UI or interrupt trading
  • Keep it professional — this is a serious tool with satisfying feedback, not a mobile game

9. Server-Side Data Architecture Planned

Migration Priority

Everything should live server-side so users can work across devices. Current localStorage-based features need migration:

  1. Session history → session_history table (user_id, date, character, trades, wins, losses, pnl, grade)
  2. Streaks → derived from session_history dates
  3. Achievements → user_achievements table (user_id, badge_id, earned_at)
  4. Iron Man → already server-side ✓
  5. Saved sessions → saved_sessions table (user_id, session_data JSON, created_at)

Schema Additions Needed

session_history: id, user_id, date, character, seed, trades, wins, losses, pnl, grade, details_json, created_at

user_achievements: id, user_id, badge_id, earned_at

user_preferences: id, user_id, key, value (for settings like sound on/off, confetti, etc.)

Design principle: The user should be able to log in from any device and immediately see their full history, streaks, Iron Man progress, and achievements. No data should live exclusively in localStorage in production.

10. What We're NOT Doing

  • Leaderboards / social comparison — this is about personal discipline, not competition
  • Pay-to-win mechanics — no purchased advantages or XP boosters
  • Lockout timers — the constraint should be psychological, not mechanical
  • AI trade suggestions — we teach process, not predictions
  • Copy trading — every trade decision must be the user's own
